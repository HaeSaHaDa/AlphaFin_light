# AlphaFin LTE 검증 규칙

## 문서 목적

이 문서는 AlphaFin LTE 프로젝트의
검증 규칙과 확인 절차를 정의한다.

목표는 다음과 같다.

- AI 구현 품질 유지
- 구조적 오류 방지
- 데이터 누락 방지
- 실험 재현 가능성 확보
- 안정적인 AI 협업 유지

현재 프로젝트는
경량형 연구 구조를 기준으로 검증한다.

---

# 검증 기본 원칙

현재 프로젝트는 다음 원칙을 유지한다.

- 구현 후 반드시 검증한다.
- 검증 없는 완료 처리를 금지한다.
- 작은 단위 검증을 우선한다.
- 샘플 기반 검증을 우선한다.
- 로그 기반 검증을 유지한다.

---

# 최소 검증 항목

모든 구현은
다음 항목을 최소 검증 기준으로 사용한다.

| 항목 | 목적 |
|---|---|
| 실행 가능 여부 | 코드 실행 확인 |
| 샘플 데이터 동작 여부 | 기본 기능 확인 |
| 로그 생성 여부 | 추적 가능성 확인 |
| 출력 결과 생성 여부 | 데이터 생성 확인 |
| 예외 처리 여부 | 실패 대응 확인 |

---

# Collector 검증 규칙

## 목적

외부 데이터 수집 안정성 검증.

## 최소 검증 항목

- API 요청 성공 여부
- 요청 실패 처리 여부
- timeout 처리 여부
- Raw Data 저장 여부
- 로그 생성 여부

## 확인 대상 예시

- pykrx 수집 결과
- OpenDART 응답
- 뉴스 수집 결과

## 현재 원칙

- 대량 요청 테스트 금지
- 샘플 기반 검증 우선
- 실패 로그 기록 필수

---

# Preprocess 검증 규칙

## 목적

전처리 안정성 검증.

## 최소 검증 항목

- 입력 데이터 존재 여부
- HTML 제거 여부
- 텍스트 정리 여부
- Chunk 생성 여부
- Metadata 생성 여부

## 확인 대상 예시

- 뉴스 텍스트
- 공시 텍스트
- Chunk 결과

---

# RAG 검증 규칙

## 목적

검색 구조 정상 동작 검증.

## 최소 검증 항목

- Embedding 생성 여부
- Vector 저장 여부
- Retrieval 결과 생성 여부
- Metadata Filtering 동작 여부
- Context 생성 여부

## 현재 원칙

- 단순 Retrieval 우선
- 과도한 최적화 금지
- 작은 샘플 기반 검증 우선

---

# LLM 분석 검증 규칙

## 목적

LLM 분석 흐름 검증.

## 최소 검증 항목

- Prompt 생성 여부
- Retrieval 연결 여부
- 응답 생성 여부
- 응답 저장 여부
- 오류 처리 여부

## 확인 대상 예시

- 뉴스 기반 분석
- 공시 기반 분석
- Retrieval 기반 응답

---

# Evaluation 검증 규칙

## 목적

실험 구조 검증.

## 최소 검증 항목

- 실험 로그 생성 여부
- Prompt 비교 가능 여부
- Retrieval 결과 저장 여부
- 실험 재현 가능 여부

---

# 데이터 검증 규칙

## 목적

데이터 품질 유지.

## 최소 검증 항목

- 빈 데이터 여부
- 중복 데이터 여부
- Metadata 존재 여부
- 날짜 형식 정상 여부
- 종목코드 정상 여부

## 현재 원칙

- Raw Data 수정 금지
- Processed Data 분리 유지
- 데이터 누락 로그 기록

---

# 로그 검증 규칙

## 목적

추적 가능한 구조 유지.

## 최소 검증 항목

- 오류 로그 생성 여부
- 실험 로그 생성 여부
- 수집 로그 생성 여부
- 구조 변경 로그 기록 여부

## 현재 예상 구조

```text
prompts/
├─ TASK-001/
├─ TASK-002/

logs/
├─ TASK-001/
├─ TASK-002/
├─ daily/
├─ issues/
├─ ai-decisions/
└─ experiments/
```

---

# TASK 검증 규칙

## 목적

작업 단위 품질 유지.

## 최소 검증 항목

- TASK 범위 준수 여부
- 관련 없는 수정 여부
- 검증 완료 여부
- 로그 기록 여부

## 현재 원칙

- 하나의 TASK는 하나의 목적만 가진다.
- 검증 없는 완료 처리 금지.
- 범위 외 수정 금지.

---

# AI 구현 검증 규칙

## 목적

AI 협업 안정성 유지.

## 현재 금지 사항

- 문서 없는 구현
- 테스트 없는 완료 처리
- 구조 임의 변경
- 과도한 abstraction
- 거대한 클래스 생성
- 하드코딩
- 예외 무시

## 현재 권장 사항

- 작은 함수 유지
- 작은 파일 유지
- 명시적 흐름 유지
- 로그 기반 디버깅 유지

---

# 현재 검증 방식

현재 프로젝트는
다음 검증 방식을 우선한다.

```text
샘플 데이터 기반 검증
→ 로그 확인
→ 출력 확인
→ 예외 확인
→ 반복 테스트
```

현재 단계에서는
대규모 자동화 테스트보다
작고 명시적인 검증을 우선한다.

---

# 향후 확장 가능 영역

향후 확장 가능 영역:

- 자동 테스트
- Benchmark 자동화
- 실험 비교 자동화
- Prompt Versioning
- Retrieval 성능 측정

단,
현재 단계에서는
복잡한 테스트 인프라를 우선하지 않는다.
